## 1. Overview & Goal
เอกสารนี้กำหนดความต้องการในการเปลี่ยนผ่านแอปพลิเคชัน JarWise จากระบบที่มี "Jars" (หมวดหมู่การใช้จ่าย/การออม) และ "Wallets" (บัญชีการชำระเงิน) แบบคงที่ (Hardcoded) ไปสู่ระบบที่ผู้ใช้สามารถจัดการเองได้แบบไดนามิก 

**เป้าหมายหลัก:** เพื่อให้ผู้ใช้มีความยืดหยุ่นในการปรับแอปพลิเคชันให้เข้ากับนิสัยทางการเงินและระบบบัญชีจริงของตนเอง เช่น การใช้ระบบซองเงิน (Envelope system), การงบประมาณแบบฐานศูนย์ (Zero-based budgeting) หรือการติดตามเป้าหมายการออมส่วนบุคคล ซึ่งจะช่วยเพิ่มความพึงพอใจและประโยชน์โดยรวมของแอปพลิเคชัน

---

## 2. User Personas
* **New User:** ต้องการตั้งค่าแอปให้สะท้อนถึงบัญชีธนาคารจริงและเป้าหมายการออมตั้งแต่เริ่มต้นใช้งาน
* **Existing User:** ต้องการปรับแต่งหมวดหมู่เริ่มต้นและเพิ่มหมวดหมู่ใหม่ๆ เช่น Jar สำหรับ "กองทุนท่องเที่ยว" หรือ Wallet สำหรับ "บัตรเครดิต"

## 4. Functional Requirements

### 4.1. Jar Management (CRUD)
* **Create/Update:**
    * `Name` (Required): ชื่อหมวดหมู่
    * `Icon` (Optional): เลือกจากรายการที่กำหนด
    * `Color` (Optional): กำหนดสีเพื่อการแสดงผลใน UI
    * `Description` (Optional): รายละเอียดเพิ่มเติม
* **Delete:** สามารถลบ Jar ได้ แต่หากมีธุรกรรมผูกอยู่ ระบบจะบังคับให้เลือก Jar ปลายทางเพื่อโอนย้ายข้อมูลก่อน

### 4.2. Wallet Management (CRUD & Archive)
* **Create/Update:**
    * `Name` (Required): ชื่อบัญชี
    * `Initial Balance` (Required): ระบุยอดเงินตั้งต้น ณ วันที่สร้าง
        * *หมายเหตุ: สำหรับลอจิกการคำนวณยอดเงินและการทำ Adjustment ให้ดูที่ Related Issue (Link 2-ways)*
* **Archive (Soft Delete):**
    * Wallet ที่ถูก Archive จะไม่ปรากฏใน Dropdown เมื่อสร้างธุรกรรมใหม่
    * Wallet และธุรกรรมที่เกี่ยวข้องจะยังคงอยู่ในระบบเพื่อความถูกต้องของรายงาน
    * ระบบต้องมีทางเลือกให้เรียกดูและ Un-archive ได้

### 4.3. Transaction Integration
* Dropdown สำหรับ "Jar" และ "Wallet" ในหน้า Add/Edit Transaction จะต้องดึงข้อมูลมาแสดงตามที่ผู้ใช้สร้างไว้
* ระบบต้องเก็บการอ้างอิง (Foreign Key) ไปยัง ID ของ Jar และ Wallet แทนการเก็บเป็น String

## 5. Non-Functional Requirements
* **Usability:** UI สำหรับจัดการต้องใช้งานง่ายและเข้าใจได้ทันที
* **Data Integrity:** การย้ายธุรกรรมเมื่อลบ Jar/Wallet ต้องทำงานแบบ Atomic (สำเร็จทั้งหมดหรือไม่ทำเลย) เพื่อป้องกันข้อมูลค้าง
* **Performance:** การโหลดรายการ Jar/Wallet ในหน้าบันทึกธุรกรรมต้องรวดเร็วและไม่ทำให้แอปหน่วง

---

## 6. User Journey
**เป้าหมาย:** สร้าง Jar ใหม่สำหรับการท่องเที่ยวและบันทึกเงินออมก้อนแรก
1.  เปิดแอปและไปที่หน้า `Settings`
2.  เลือก "Manage Jars & Wallets" และไปที่แถบ "Jars"
3.  กดปุ่ม "Add New Jar" (+)
4.  กรอกชื่อ "Hawaii Vacation", เลือกไอคอนต้นมะพร้าว, และเลือกสีเขียว
5.  บันทึกข้อมูลและเห็น Jar ใหม่ปรากฏในรายการ
6.  ไปหน้าหลักเพื่อเพิ่มธุรกรรมใหม่ เลือก Jar "Hawaii Vacation" และเลือก Wallet "Main Savings"
7.  ใส่ยอดเงิน $200 และบันทึกสำเร็จ

## 7. Specification by Example (SBE)

### Scenario 1: การสร้างและใช้งาน Wallet ใหม่
* **Given:** ผู้ใช้มี Wallet "Main Chequing" ($1,500) และ "Visa Card" (-$250)
* **When:** สร้าง Wallet ใหม่ชื่อ "PayPal" พร้อมยอดเริ่มต้น $100
* **Then:** รายการ Wallet จะเพิ่ม "PayPal" ($100) เข้ามา
* **And When:** บันทึกค่าอาหาร $25 โดยใช้ Wallet "PayPal"
* **Then:** ยอดเงินใน "PayPal" จะเหลือ $75

### Scenario 2: การลบ Jar ที่มีธุรกรรมอยู่
* **Given:** มี Jar "Old Savings Goal" ซึ่งมีธุรกรรมผูกอยู่ 2 รายการ
* **When:** ผู้ใช้พยายามลบ Jar "Old Savings Goal"
* **Then:** ระบบแสดงข้อความ *"Jar นี้มี 2 ธุรกรรมเกี่ยวข้อง กรุณาเลือก Jar ใหม่เพื่อย้ายข้อมูลก่อนลบ"*
* **And When:** ผู้ใช้เลือกย้ายข้อมูลไปที่ Jar "Fun Money" และยืนยัน
* **Then:** Jar "Old Savings Goal" ถูกลบ และธุรกรรมทั้ง 2 จะไปปรากฏภายใต้ Jar "Fun Money"

### Scenario 3: การ Archive Wallet
* **Given:** มี Wallet "Old Bank Account" ที่ปิดบัญชีไปแล้วและมีธุรกรรมเก่า 42 รายการ
* **When:** ผู้ใช้กด Archive บัญชีนี้
* **Then:** ระบบยืนยันการ Archive และ Wallet นี้จะไม่แสดงในหน้าทำรายการใหม่
* **And:** เมื่อดูรายงานย้อนหลัง ธุรกรรม 42 รายการเดิมจะยังคงแสดงผลถูกต้องตาม Wallet เดิม

## 8. Data Migration Strategy
ระบบจะทำการย้ายข้อมูลอัตโนมัติเมื่อผู้ใช้เปิดแอปเวอร์ชันใหม่ครั้งแรก:
1.  **Schema Update:** เพิ่มตาราง `jars` และ `wallets` และปรับปรุงตาราง `transactions` ให้ใช้ Foreign Key
2.  **Logic:** สร้างรายการในตารางใหม่ตามค่า Hardcoded เดิม (เช่น 'NEC', 'SAVINGS') สำหรับผู้ใช้แต่ละราย
3.  **Update:** อัปเดตธุรกรรมเดิมทั้งหมดให้เชื่อมโยงกับ ID ของ Jar และ Wallet ที่สร้างขึ้นใหม่
4.  **Impact:** กระบวนการนี้ต้องไม่ส่งผลกระทบต่อข้อมูลเดิมและผู้ใช้ต้องไม่รู้สึกถึงความเปลี่ยนแปลงของข้อมูลประวัติ

---

## 9. Out of Scope
* การจัดเรียง (Sorting) ลำดับ Jar และ Wallet ด้วยตนเอง
* การตั้งงบประมาณ (Budgeting) หรือเป้าหมาย (Targets) ลงบน Jar โดยตรง
* การแชร์ Jar หรือ Wallet ร่วมกับผู้ใช้อื่น
* การทำ Jar ย่อยหรือหมวดหมู่ซ้อน (Nested Jars)
