### **คู่มือการตรวจสอบด้วยตนเอง (Manual Verification Guide)**

คู่มือนี้มีวัตถุประสงค์เพื่อทดสอบความทนทานของระบบ (Robustness) และความถูกต้องของข้อมูล (Data Integrity) ในสถานการณ์ที่ซับซ้อนตามที่ระบุใน Code Review

#### **🧪 กรณีทดสอบที่ 1: การลบ Wallet ที่มี Jar ลูก (Hierarchy Deletion)**

**เป้าหมาย:** ตรวจสอบว่าระบบจัดการข้อมูลอย่างถูกต้องเมื่อผู้ใช้พยายามลบ "Wallet" (บัญชีหลัก) ที่มี "Jar" (บัญชีย่อย) และธุรกรรมอยู่ข้างใน เพื่อป้องกันข้อมูลสูญหาย

**ขั้นตอนการทดสอบ:**
1.  **สร้างโครงสร้างข้อมูล:**
    *   สร้าง Wallet ใหม่ชื่อ "เงินลงทุน".
    *   ภายใน Wallet "เงินลงทุน", สร้าง Jar ย่อย 2 อัน ชื่อ "หุ้น" และ "กองทุนรวม".
    *   เพิ่มธุรกรรมเข้าไปใน Jar "หุ้น" (เช่น ซื้อหุ้น 5,000 บาท) และ Jar "กองทุนรวม" (เช่น ซื้อกองทุน 3,000 บาท).
2.  **พยายามลบ Wallet แม่:**
    *   ไปที่หน้าจัดการ Wallet และเลือกที่จะลบ Wallet "เงินลงทุน".

**ผลลัพธ์ที่คาดหวัง (อย่างใดอย่างหนึ่งต่อไปนี้):**
*   **ป้องกันการลบ:** ระบบแสดงหน้าต่างข้อผิดพลาด (Error) แจ้งว่า "ไม่สามารถลบ Wallet ได้ เนื่องจากมี Jar และธุรกรรมอยู่ข้างใน" และการลบไม่สำเร็จ
*   **เตือนและยืนยัน:** ระบบแสดงหน้าต่างคำเตือนที่ชัดเจน (Warning) เช่น "การลบ Wallet นี้จะลบ Jar และธุรกรรมทั้งหมดที่เกี่ยวข้องอย่างถาวร คุณแน่ใจหรือไม่?" และต้องการการยืนยันที่ชัดเจนจากผู้ใช้ก่อนดำเนินการต่อ
*   **ย้ายข้อมูล:** ระบบแสดงหน้าต่างให้เลือกว่าจะย้าย Jar และธุรกรรมทั้งหมดไปยัง Wallet อื่นที่มีอยู่ก่อน แล้วจึงลบ Wallet "เงินลงทุน" ที่ว่างเปล่า
*   **สิ่งที่ไม่ควรเกิดขึ้น:** Wallet ถูกลบไปเงียบๆ โดยไม่มีคำเตือน และทำให้ธุรกรรมใน Jar "หุ้น" และ "กองทุนรวม" หายไปจากระบบ หรือกลายเป็นข้อมูลกำพร้า (Orphaned Data)

#### **🧪 กรณีทดสอบที่ 2: การนำเข้าข้อมูลที่เสียหาย (Corrupted Data Migration)**

**เป้าหมาย:** ทดสอบว่าฟังก์ชันการย้ายข้อมูลจากแอป Money Manager (#65) สามารถรับมือกับไฟล์ `.mmbak` ที่มีข้อมูลเสียหายหรือไม่สมบูรณ์ได้ โดยไม่ทำให้แอปแครชหรือข้อมูลใหม่เสียหาย

**ขั้นตอนการทดสอบ:**
1.  **เตรียมไฟล์ทดสอบ:**
    *   หาไฟล์ `.mmbak` ที่ถูกต้องและสมบูรณ์มาหนึ่งไฟล์
    *   เปิดไฟล์เพื่อแก้ไข (อาจต้องใช้เครื่องมือที่เหมาะสม) และจงใจทำให้ข้อมูลเสียหาย เช่น ลบข้อมูลการประกาศ Wallet บางอันออกไป แต่ยังคงเหลือธุรกรรมที่อ้างอิงถึง Wallet นั้นอยู่
2.  **เริ่มกระบวนการนำเข้าข้อมูล:**
    *   ในแอป JarWise, เริ่มกระบวนการนำเข้าข้อมูลจาก Money Manager และเลือกไฟล์ `.mmbak` ที่แก้ไขไว้

**ผลลัพธ์ที่คาดหวัง:**
*   **แอปไม่แครช:** กระบวนการนำเข้าไม่ควรทำให้แอปพลิเคชันหยุดทำงาน
*   **การรายงานข้อผิดพลาด:** ระบบควรแสดงรายงานข้อผิดพลาดที่ชัดเจนให้ผู้ใช้ทราบว่า "พบข้อมูลที่ไม่สมบูรณ์ในไฟล์และไม่สามารถนำเข้าบางรายการได้"
*   **การจัดการข้อมูลที่ผิดพลาด:** ธุรกรรมที่ผิดพลาด (ที่อ้างอิงถึง Wallet ที่ไม่มีอยู่) ควรถูกข้ามไป (Skip) หรือถูกนำเข้าไปเก็บในหมวดหมู่พิเศษเช่น "รายการที่ยังไม่แก้ไข" (Unresolved) เพื่อให้ผู้ใช้ตรวจสอบภายหลัง
*   **สิ่งที่ไม่ควรเกิดขึ้น:** แอปแครช หรือข้อมูลที่ถูกต้องอื่นๆ ในไฟล์ถูกนำเข้าไปอย่างไม่ถูกต้อง หรือทำให้ฐานข้อมูลของผู้ใช้เสียหาย

#### **🧪 กรณีทดสอบที่ 3: ความขัดแย้งของการซิงค์ข้อมูลระหว่างอุปกรณ์ (Sync Conflict)**

**เป้าหมาย:** ทดสอบการจัดการ Conflict Resolution ของระบบคลาวด์ซิงค์ (#32) เมื่อเกิดการเปลี่ยนแปลงข้อมูลที่ขัดแย้งกันระหว่างอุปกรณ์สองเครื่อง ขณะทำธุรกรรมโอนย้าย (#71)

**ขั้นตอนการทดสอบ:**
1.  **เตรียมอุปกรณ์:**
    *   ล็อกอินเข้าสู่ระบบ JarWise ด้วยบัญชีเดียวกันบนอุปกรณ์ 2 เครื่อง (เช่น เว็บเบราว์เซอร์ และ โทรศัพท์ Android)
    *   ตรวจสอบให้แน่ใจว่าทั้งสองเครื่องมีข้อมูล Wallet A และ Wallet B ที่ตรงกัน
2.  **สร้าง Conflict:**
    *   **บนอุปกรณ์ Android:** ทำธุรกรรม "โอนเงิน" (Transfer) จาก Wallet A ไปยัง Wallet B
    *   **อย่างรวดเร็ว (ก่อนที่การซิงค์จะเสร็จสมบูรณ์):** **บนเว็บเบราว์เซอร์**, ให้ทำการ "เปลี่ยนชื่อ" หรือ "ลบ" Wallet B
3.  **รอให้ระบบซิงค์:**
    *   ปล่อยให้ทั้งสองอุปกรณ์เชื่อมต่ออินเทอร์เน็ตและทำการซิงค์ข้อมูล

**ผลลัพธ์ที่คาดหวัง:**
*   **ไม่มีข้อมูลสูญหาย:** เงินไม่ควร "หาย" ไปจากระบบ (เช่น ถูกหักจาก Wallet A แต่ไม่เข้าที่ไหนเลย)
*   **สถานะที่สอดคล้องกัน:** ระบบควรจัดการความขัดแย้งนี้อย่างใดอย่างหนึ่ง เช่น:
    *   ธุรกรรมการโอนเงินล้มเหลว และเงินถูกคืนกลับไปยัง Wallet A พร้อมแจ้งเตือนผู้ใช้
    *   ระบบแจ้งเตือนผู้ใช้ว่าเกิดความขัดแย้ง และให้ผู้ใช้ตัดสินใจว่าจะทำอย่างไร (เช่น เลือก Wallet ปลายทางใหม่)
*   **สิ่งที่ไม่ควรเกิดขึ้น:** ข้อมูลของทั้งสองอุปกรณ์อยู่ในสถานะที่ไม่ตรงกัน (Inconsistent State) เช่น อุปกรณ์หนึ่งแสดงว่าเงินถูกโอนสำเร็จ แต่อีกอุปกรณ์หนึ่งไม่มี Wallet ปลายทางนั้นอยู่แล้ว
